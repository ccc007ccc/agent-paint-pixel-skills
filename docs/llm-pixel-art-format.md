# LLM 像素画源格式思路

## 目标

让 LLM 能稳定地“手动画素画”，同时支持不同画布比例、分辨率、图层和后续局部修改。

核心原则：

- LLM 负责构图、调色板、图层内容和局部修改。
- 程序负责校验、解析、合成、预览、导出 RGBA/PNG。
- 不让 LLM 直接输出大段 RGBA 数组。

## 推荐格式

最适合 LLM 的基础格式是：

```txt
HEX/RGB 调色板 + 单字符索引网格
```

示例：

```txt
canvas: 16x16

palette:
. = transparent
K = #171717
R = #d93636
r = #ff6b6b
W = #ffffff
S = rgba(0,0,0,0.35)

sprite:
................
................
.....KKKKKK.....
....KRRRRRRK....
...KRRrrrrRRK...
...KRrrWWrrRK...
...KRrrrrrrRK...
....KRRRRRRK....
.....KKKKKK.....
................
................
................
................
................
................
................
```

这样 LLM 只需要控制每个字符的位置。每个字符代表一个像素，`.` 固定代表透明。

## 为什么不用直接 RGBA

直接 RGBA 数组虽然理论上可行，但不适合 LLM：

- 数字重复量太大。
- 容易漏像素、错行、错列。
- 人类很难检查形状。
- 大画布和多图层时 token 成本迅速膨胀。

例如 64x64 就有 4096 个像素。如果每个像素写 `[r,g,b,a]`，LLM 很容易产生格式错误。索引网格则更接近人类手动画素画。

## 图层格式

复杂像素画应该拆成图层，而不是一张图一次性画完。

建议的最小图层字段：

```json
{
  "name": "base",
  "visible": true,
  "opacity": 1,
  "rows": [
    "................",
    ".....RRRR.......",
    "....RRrrRR......",
    "................"
  ]
}
```

空图层也是合法的，适合占位、临时隐藏方案或执行 `clear_layer` 之后继续追加局部块：

```json
{
  "name": "paint_over"
}
```

常见图层顺序从上到下排列，第一行是最顶层，和 Photoshop 图层面板一致：

```txt
effects
details
eyes
highlight
mid_shadow
dark_shadow
secondary_color
base_color
outline
silhouette
shadow
background
```

程序从数组末尾的底层开始合成到数组开头的顶层，透明像素跳过，不透明或半透明像素根据 alpha 和 opacity 混合。

## 局部块

当画布达到 64x64、128x128 或更高时，不建议让 LLM 一次输出整张图。应该支持局部块。

示例：

```json
{
  "layer": "eyes",
  "x": 10,
  "y": 8,
  "width": 12,
  "height": 4,
  "rows": [
    "..WW....WW..",
    ".WKKW..WKKW.",
    "..KK....KK..",
    "............"
  ]
}
```

局部块的好处：

- LLM 输出更短。
- 适合迭代修改。
- 可以像 Photoshop 一样只改某个图层的某个区域。
- 便于做撤销、差异记录和版本对比。

## 画布规模建议

```txt
16x16: 可以一次完整输出。
32x32: 可以完整输出，建议使用图层。
64x64: 建议图层 + 局部块。
128x128+: 必须分块、分层或使用绘图命令。
```

## 可选绘图命令

除了索引网格，也可以支持命令式编辑。命令适合做规则形状、批量修改和局部修正。

示例：

```json
[
  { "op": "fillRect", "layer": "shadow", "x": 10, "y": 18, "w": 12, "h": 3, "color": "S" },
  { "op": "setPixel", "layer": "eyes", "x": 15, "y": 9, "color": "W" },
  { "op": "line", "layer": "outline", "from": [8, 12], "to": [20, 12], "color": "K" }
]
```

命令式编辑不应该完全替代索引网格。更合理的组合是：

- 大体形状、角色轮廓、局部细节用索引网格。
- 批量填充、移动、擦除、描边、镜像用命令。

## 校验规则

程序必须在渲染前校验 LLM 输出：

- 画布宽高是否有效。
- 每个 rows 的行数是否正确。
- 每行字符数是否正确。
- 是否使用了未定义的调色板索引。
- 图层或局部块是否越界。
- opacity 是否在 0 到 1 之间。
- 颜色是否能解析为 transparent、HEX、RGB 或 RGBA。

校验失败时不要直接渲染，应该把错误反馈给 LLM，让它只修复错误片段。

## 建议工作流

1. 用户输入主题、尺寸、风格、复杂度。
2. LLM 生成画布、调色板和图层计划。
3. LLM 分层输出索引网格或局部块。
4. 程序校验格式。
5. 程序合成图层并渲染预览。
6. 用户或 LLM 根据预览提出修改。
7. LLM 输出局部块或命令进行增量编辑。
8. 程序再次校验、合成、导出 PNG/RGBA。

## 结论

最适合 LLM 的像素画表示方式是：

```txt
HEX/RGB 调色板 + 单字符索引网格 + 图层 + 局部块 + 自动校验 + 预览迭代
```

LLM 不应该直接维护最终 RGBA 数据。最终 RGBA 应该由程序从像素画源格式稳定生成。
